Node.js
- 브라우저 밖에서도 자바스크립트를 실행할 수 있도록 만든 서버 즉 자바스크립트 "실행 환경"
- 구글의 V8 엔진을 기반으로 동작
- 웹 서버, API 서버, 실시간 채팅, 스트리밍 서비스 등 활용
- npm(Node Package Manager)을 통해 수많은 오픈소스 패키지를 쉽게 사용할 수 있어 빠른 개발이 가능 (pip와 비슷)
- 프론트엔드와 동일한 언어(Javascript)를 사용한다는 점에서 풀스택 개발에 매우 유리한 환경

프론트엔드, 백엔드 역할 차이
- 프론트엔드는 사용자가 직접 보고 상호작용하는 화면(UI)를 구현하는 영역. HTML/CSS/Javascript를 사용해 버튼, 입력창, 화면 레이아웃, 애니메이션 등 사용자 경험을 담당
- 백엔드는 사용자의 요청을 받아 데이터를 처리하고, 데이터베이스와 연결하여 저장/조회/수정/삭제 등의 로직을 수행하는 서버 측 영역
- https://nodejs.org/ko
- 설치 후 터미널에서 node -v로 버전 확인

패키지 매니저
- Node.js 환경에서 필요한 라이브러리(패키지)를 설치, 관리, 업데이트할 수 있게 도와주는 도구
- npm: node.js 설치 시 자동으로 같이 설치됨. 패키지 매니저 역할을 함

package.json⭐
- Node.js 프로젝트의 정보를 담고 있는 핵심 파일
- 패키지에 대한 설명서(프로젝트 이름, 버전, 필요 패키지 목록, 실행 스크립트 등 다양한 정보를 담고 있음)
{
  "name": "7_nodejs",       프로젝트 이름(소문자로 작성, 공백x)
  "version": "1.0.0",       프로젝트 버전
  "description": "",        간단한 설명
  "license": "ISC",         사용 라이선스
  "author": "",             만든 사람 이름
  "type": "commonjs",       프로젝트 방식
  "main": "index.js",       진입점 파일
  "scripts": {              터미널에서 사용할 실행 명령어(예: npm start, npm run dev)
    "test": "echo \"Error: no test specified\" && exit 1"
  }
}


웹 서버
- 사용자의 브라우저 요청을 받아 웹 페이지, 이미지, CSS, Javascript 같은 파일을 전달해주는 서버 프로그램 또는 시스템
- 사용자가 주소(URL)를 입력하면 웹서버는 해당 요청을 처리하고 필요한 파일이나 데이터를 응답으로 보내며, 이 과정을 통해서 브라우저에 웹 사이트 화면이 표시됨

모듈
- 모듈은 프로그램을 기능별로 나누어 독립적인 파일이나 단위로 구성한 것으로, 코드의 재사용성과 유지보수를 쉽게 해주는 구조
- (node.js는 모듈 단위로 개발을 함)
- 특정 기능(계산, 인증, 데이터 처리 등)을 하나의 파일로 분리해두면, 다른 파일에서 필요할 때 불러와 사용할 수 있음
- require() 또는 import를 통해 모듈을 가져올 수 있음

CommonJS vs ES6 모듈 차이
CommonJS
    - node.js 기본 방식
    - 모듈 가져오기: require('파일명')
    - 모듈 내보내기: module.exports 또는 exports.이름
    - 확장자: .js

ES6
    - ES6 지원하는 브라우저 또는 node.js 환경
    - 모듈 가져오기: import { 이름 } from '파일명'
    - 모듈 내보내기: export, export default
    - 확장자: .mjs

MongoDB
- NoSQL 데이터베이스 시스템 중 하나로, 문서 지향 데이터베이스
- 관계형 데이터베이스와는 다르게 데이터를 테이블이 아니라 JSON 스타일의 BSON(Binary JSON) 형식의 문서로 저장

문서 지향 데이터베이스

스키마 없음
- 관계형 데이터베이스와 달리 MongoDB는 데이터베이스의 스키마를 명시적으로 정의하지 않음
- 동적인 스키마를 사용하여 데이터 모델을 유연하게 변경할 수 있도록 함

JSON Web Token(JWT)
- 서버와 클라이언트 사이에서 인증 정보를 안전하게 전달하기 위한 컴팩트하고 독립적인 토큰
- JWT는 세 부분(헤더, 페이로드, 서명)으로 구성되어 있음
  - 헤더: 토큰이 어떤 알고리즘으로 암호화되었는지, 어떤 형식인지 알려주는 부분 (정해진 키값)
    {
      "alg": "HS256",   // 서명 알고리즘 (예: HMAC SHA-256 등 암호화 알고리즘)
      "typ": "JWT"      // 토큰 타입
    }
  - 페이로드: 실제 데이터(사용자 정보, 권한 등)가 담기는 부분 (내가 만들어서 보내는 것)
    {
      "userid": "apple",
      "role": "user",
      "iat": 123456789,   // 토큰 발급 시간
      "exp": 123456999    // 토큰 만료 시간
    }
  - 서명: 헤더와 페이로드가 변조되지 않았는지 검증하는 부분
    HMACSHA256(
      Base64UrlEncode(header) + "." + Base64UrlEncode(payload), secret
    )
      - '인코딩된 Header + . + 인코딩된 payload'를 하나의 문자열로 합침
      - 서버만 알고 있는 비밀키를 사용해서 서명함
      - 서버는 서명을 검증해서 토큰이 위조되지 않았음을 확인할 수 있음
      - 비밀키를 아는 서버만 서명을 검증할 수 있음
        (-> 비밀키는 밖으로 노출되면 안됨)

bcrypt
- 비밀번호와 같은 민감한 데이터를 안전하게 저장하기 위해 사용하는 해시 알고리즘
- 일반적인 암호화와 다르게 bcrypt는 입력된 값을 복호화할 수 없음
- 오직 비교만 가능하게 설계되어 있어 비밀번호 유출 위험을 크게 줄여줌

  salt: 해시하기 전에 추가하는 무작위 문자열
  hash: salt를 포함해서 만든 결과값
  rounds: 해시 연산을 몇 번 반복할지 정하는 숫자(예: 10번 반복)

  예)
    1111 -> 14253a4bf2652d
    1234567 -> 05834818957d2a1849190e15b
    1111 -> 14253a4bf2652d (같은 값으로 암호화)

    apple / 1111 -> 14253a4bf2652d
